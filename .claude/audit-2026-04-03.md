## Phase 1: Local Repository Analysis

### Project Identity

**Ralphglasses** is a command-and-control TUI + bootable thin client for parallel multi-LLM agent fleets. Two deliverables: (1) a cross-platform Go binary with k9s-style TUI, and (2) a bootable Linux thin client (Manjaro + Sway/Hyprland, 7-monitor, dual-NVIDIA-GPU).

### Architecture Highlights

1. **Provider dispatch pattern** — `buildCmdForProvider()` + `normalizeEvent()` abstracts Claude/Gemini/Codex CLI subprocess management
2. **Deferred tool loading** — Core 10 tools always loaded; 136 namespace tools load on demand (85% context token reduction)
3. **6-layer MCP middleware stack** — Concurrency → Trace → Timeout → Instrumentation → EventBus → Validation
4. **Self-improvement pipeline** — 5 subsystems: Reflexion (failure injection), Episodic Memory (trajectory recall), Cascade Router (cheap-then-expensive), Curriculum Sorter (difficulty ordering), Auto-Optimizer (config tuning)
5. **R&D cycle engine** — Proposed → Baseline → Execute → Observe → Synthesize → Complete state machine
6. **Fleet coordination** — A2A dispatch, blackboard CAS, bandit-based provider selection (UCB1 + Thompson sampling + contextual)

## Phase 2: GitHub Research Findings

### Category 1: Ralph Loop Pattern (3 findings)

1. **snarktank/ralph** — The canonical open-source ralph loop. Shell script running Claude/Amp repeatedly until PRD items pass. Uses `<promise>COMPLETE</promise>` exit sentinel. Active.
2. **Pattern went mainstream** — "Ralph Wiggum Loop" is now a Cursor plugin, documented at ASDLC.io, reports 60% rework reduction.
3. **Claude Code absorbed it as `/loop`** — Session-level scheduler, default 10m interval. Simpler than ralphglasses' loop tools but validates the pattern. Commoditization threat.

### Category 2: Claude Code Orchestration (3 findings)

4. **Claude Code Agent Teams** — Experimental built-in (v2.1.32+). One lead, teammates with independent context. Direct messaging. Claude-only, no resume for teammates.
5. **claude-fleet** (github.com/sethdford/claude-fleet) — TypeScript. Specialized agents (Lead, Scout, Kraken, Oracle, Critic, Architect). Blackboard context sharing. Claude-only.
6. **Ruflo** (github.com/ruvnet/ruflo) — ~500K downloads. 100+ specialized agents, 8 categories. "Hive Mind" queen-led hierarchical coordination. Claude + Codex.

### Category 3: Multi-LLM Agent Frameworks (3 findings)

7. **Composio Agent Orchestrator** (github.com/ComposioHQ/agent-orchestrator) — Agent-agnostic. Auto-CI-fix handling, merge conflict resolution, review comment routing. Built 40K LOC + 3288 tests in 8 days. Key feature: auto-CI-fix.
8. **Overstory** (github.com/jayminwest/overstory) — 11 pluggable runtime adapters. SQLite mail system for inter-agent communication. STEELMAN.md risk analysis.
9. **Industry consensus** — LangGraph (directed graphs), CrewAI (role-based), OpenAI Agents SDK (handoffs), AutoGen/AG2 (GroupChat), Google ADK (hierarchical), Claude SDK (tool-use chains). Multi-LLM orgs report 60% lower costs.

### Category 4: MCP Server Frameworks in Go (3 findings)

10. **Official MCP Go SDK** (github.com/modelcontextprotocol/go-sdk) — v1.4.1 stable with compatibility guarantee. DNS rebinding CVE fixed (CVE-2026-34742). OAuth support. Google collaboration.
11. **mcp-go** (github.com/mark3labs/mcp-go) — Community Go MCP, high-level. Complete core spec. Lacks middleware-first design.
12. **gopls MCP support** — Go language server exposes functionality via MCP natively. Claude Code Go sessions get gopls tools automatically.

### Category 5: A2A Protocol (4 findings)

13. **Official A2A Go SDK** (github.com/a2aproject/a2a-go) — v1.0 spec. High-level server (`a2asrv`) and client (`a2aclient`). REST + JSON-RPC + gRPC. Linux Foundation governed. 150+ partners.
14. **tRPC A2A Go** (github.com/trpc-group/trpc-a2a-go) — Alternative from Tencent's tRPC team.
15. **Google ADK Go 1.0** (github.com/google/adk-go) — Native OTel integration, self-healing plugins, YAML agent definitions, A2A support.
16. **MCP + A2A complementary** — Industry consensus: MCP for tool/context, A2A for agent-to-agent. Both Linux Foundation. ralphglasses implementing both is the standard pattern.

### Category 6: TUI Agent Dashboards (5 findings)

17. **Agent Deck** (github.com/asheshgoplani/agent-deck) — CLOSEST COMPETITOR. Go + BubbleTea. Claude, Aider, Gemini CLI, Codex. Session forking with full conversation history. "Conductors" (persistent orchestrator sessions). Real-time cost tracking.
18. **TmuxCC** (github.com/nyanko3141592/tmuxcc) — Rust TUI for AI agents in tmux. Lighter-weight.
19. **Conduit** (github.com/conduit-cli/conduit) — Multi-agent TUI with tabbed interface. **Web UI with full TUI feature parity** — feature ralphglasses lacks.
20. **Pertmux** (github.com/rupert648/pertmux) — Links GitLab/GitHub MRs to branches/worktrees/tmux/agents.
21. **Kagent** (github.com/kagent-dev/kagent) — Kubernetes-native AI agent framework with k9s-inspired BubbleTea TUI. CNCF-bound.

### Category 7: Prompt Enhancement (3 findings)

22. **DSPy + GEPA** (github.com/stanfordnlp/dspy) — LM reflection on trajectories. MIPROv2 Bayesian optimization. BetterTogether prompt + weight optimization. GEPA reflection aligns with ralphglasses' self_improve.
23. **Promptomatix** (Salesforce) — Cost-aware prompt optimization. Quality + efficiency balance.
24. **Evidently** — Automated prompt optimization via search-based and example-based methods.

### Category 8: Cost Optimization (3 findings)

25. **Bifrost** (github.com/maximhq/bifrost) — Go AI gateway. 20+ providers, single API. 11μs overhead at 5K RPS. 4-tier budget hierarchy. Native Prometheus. Apache 2.0. Could integrate as gateway layer.
26. **Langfuse** (langfuse.com) — Most adopted open-source LLM observability. MIT, self-hostable. Acquired by ClickHouse. Could complement internal cost tracking.
27. **Industry consensus** — Model routing + caching delivers 40-60% savings. Full optimization stack: 70-90% reduction. Hidden costs (embeddings, retries, logging) = 20-40% of total.

### Category 9: Bootable AI Thin Clients (1 finding)

28. **BLUE OCEAN — No competitors found.** Exhaustive search. Existing thin client ecosystem (ThinStation, Porteus Kiosk, LEAF OS) is entirely RDP/VDI/kiosk. Nobody builds bootable Linux for AI agent workstations. ralphglasses' distro/ is genuinely novel.

### Category 10: Claude Code Recent Features (3 findings)

29. **MCP Tool Search / Deferred Loading** — Shipped Jan 2026. Detects >10K token tool lists, injects Tool Search instead. 85-95% context reduction. Validates ralphglasses' deferred loading approach.
30. **Hook ecosystem maturation** — "defer" permission in PreToolUse hooks, Elicitation hooks, PermissionDenied hook, MCP OAuth, named subagents.
31. **Rendering & performance** — `CLAUDE_CODE_NO_FLICKER=1` alt-screen rendering. BubbleTea v2 Mode 2026 reduces tearing. 30% faster.

### Category 11: Competing Agent Orchestrators (6 findings)

32. **Claude Agent SDK** (5.1K stars) — Python/TS. Could make subprocess management obsolete for Claude sessions. No Go SDK.
33. **OpenAI Agents SDK** — Python. Handoff pattern. Guardrails, tracing. Not Go-native.
34. **Google ADK Go** — Native Go. OTel integration, A2A support. Closest to ralphglasses' architectural philosophy.
35. **OpenCode** (~95-120K stars) — Go + BubbleTea. Single-session coding tool. Validates Go + BubbleTea for AI TUI but different niche (not fleet orchestrator).
36. **Aider** — CLI pair programming. Architect/editor two-model pattern. No fleet management.
37. **Claw Code** — Claude Code fork/competitor. Growing community.

### Strategic Takeaways

**Competitive position:** ralphglasses uniquely combines multi-provider orchestration + Go TUI + bootable thin client + 146-tool MCP server + self-improvement pipeline. No single competitor covers all five pillars.

**Top 3 threats:**
1. Claude Code absorbing ralph loop as /loop + Agent Teams maturing (commoditization of core pattern)
2. Agent Deck matching the Go + BubbleTea + multi-provider feature set
3. Python/TS Agent SDKs making Go subprocess approach feel second-class

**Top 5 opportunities:**
1. Adopt official A2A Go SDK — become spec-compliant before competitors
2. Ship the bootable thin client — zero-competition differentiator
3. Integrate Bifrost as gateway layer — 20+ providers with 11μs overhead
4. Add session forking (Agent Deck's killer feature)
5. Complete learned router (existing bandit infrastructure ready, just needs training pipeline)

### Sequencing Logic

1. **mcp-go migration FIRST** — Every new handler increases migration cost. Official SDK has CVE fix. Foundation for MCP Tasks, OAuth, Streamable HTTP.
2. **A2A compliance EARLY** — Existing code is 70% there. Small effort, big interop payoff. Do alongside mcp-go migration.
3. **Session forking and auto-CI-fix together** — Both extend session lifecycle. Forking needs worktree branching (exists). Auto-CI-fix needs signal classification (exists).
4. **Learned router after OTel** — Need observation data pipeline before training. OTel provides the data.
5. **Web UI after BubbleTea v2** — Want clean rendering before exposing via browser.
6. **ISO pipeline can run anytime** — Independent of all other work. But high strategic value.

### Risk Factors

1. **mcp-go migration scope** — 118 files is massive. Mitigation: adapter layer (`internal/mcpcompat/`) provides gradual migration path. Run both SDKs during transition.
2. **BubbleTea v2 breaking changes** — Already on v2 imports but some patterns may be v1-style. Mitigation: golden tests catch rendering regressions.
3. **Claude Agent SDK has no Go version** — Can't natively integrate. Mitigation: enhance subprocess management instead; watch for Go SDK announcement.
4. **Thin client hardware dependency** — Tested only on ProArt X870E target. Mitigation: QEMU smoke test in CI, hardware profile abstraction.
5. **Self-improvement feedback loops** — 5 subsystems need tighter coupling. Mitigation: incremental — wire one cross-subsystem feedback per sprint.

### Verification Strategy (End-to-End)

For each phase, verify:

1. **Build gate:** `go build ./...` clean
2. **Test gate:** `go test ./... -count=1 -race` passes
3. **Lint gate:** `go vet ./...` clean
4. **Coverage gate:** Maintain >83% overall (current baseline)
5. **MCP gate:** All 146+ tools callable and return valid responses
6. **TUI gate:** All 19 views render without panics on standard terminal sizes
7. **Integration gate:** `go test -tags=integration ./...` passes
8. **Benchmark gate:** No >10% regression on hot paths
9. **ISO gate (Phase A4+):** QEMU smoke test passes in CI
